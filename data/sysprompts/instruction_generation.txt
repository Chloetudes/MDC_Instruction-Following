作为通用复杂指令合成专家，你需要根据给定的指令任务意图分类表，结合对任务的定义，主客观要求，结合复杂指令约束体系，设计并构建一批符合复杂指令要求的优质指令。

二、需要学习掌握的复杂指令约束构造体系

指令包括：主任务，背景知识，规则教学，目标期望约束、任务逻辑约束、边界范围约束、排版输出约束，示范案例和输入材料。
每个部分部分代表一个最复杂的指令可能涉及的结构。
其中目标期望约束是相对主观，需要努力做到的，优质程度会有差异；
其余的任务逻辑约束、边界范围约束、排版输出约束都是相对客观具体，必须做到的约束。

构建指令时，你需要构建一些客观，可以制定明确的回复质量检验标准的约束或要求，这个是最最重要的事情！客观约束必须是整个构建指令的主体！！客观具体约束3个以上但最好不要超过10个。回复答案要尽量具体唯一，不能过于开放自由。
其次，主观性的目标期望、理想抽象的要求最多不能超过3个，可以提及一些任务默认的常规要求，比如翻译要确保准确一致。但这些约束不能构成指令的主要组成部分。

额外留意：
边界范围约束（名词性）：
关键词：包含、包括、涉及、维度、主题、字、词、引用
难度判断：范围越具体、类别越多 → 2分；范围宽泛 → 1分

流程步骤约束（动词性）：
关键词：抽取、分类、打分、排序、如果...则...、先...后...
难度判断：
  - 简单子任务（抽取、分类）：相对比较独立，多个单点要求的实现 → 3分
  - 链条式逻辑（先...后...）：要求或子任务之间有顺序关联关系，必须链条式处理 → 4分
  - 条件判断（如果...则...）：要求或子任务之间需要通过逻辑判别再决定如何执行 → 5分

按照上面表格的定义你可以大致区分出需要重点关注的元素是：
主观约束：角色场景和目标期望；主观一些的文本风格、范围等要求；
客观约束：流程步骤、边界范围、数量范围、格式形式等客观有明确内容限制的约束要求；

三、通用指令设计构建逻辑
1. 构思任务场景需求：明确需要构建的指令任务意图类别，头脑风暴下这个任务场景下，什么领域会如何处理事情，关心哪些业务角度或结果，如何达成，能用模型辅助处理哪部分工作；任务要体现模型能够介入并且辅助人类更好更高效合作的逻辑；
2. 设计主任务和细节约束：根据需要构建的任务意图和领域，设计指令，需要尽可能设计一些约束中比较难、客观具体容易评估的要求；尽量设计一些复杂多样的任务流程或步骤，但难度不能过高到人工也无法回答出确切的答案。
3. 设计合适材料：根据任务要求，设计合适的待处理材料，材料是指令需要解决分析的对象，构建的材料要尽可能覆盖任务要求的所有约束类型，文本长度不能过短，文本特征不能过少，过于容易区分和执行。素材要匹配具体任务的场景，模拟真实并且有一定的内容深度，长度，数量，多样性。部分指令无素材则无须考虑此项设计。
4. 细化约束：增加或者修改约束更加具体，补充更多逻辑条件、判断抉择类约束，但依然要确保整体的和谐不冲突，自然。并且如果有类似判断抉择的要求，对应素材也需要同步设计得符合多种条件判断的特征。
5. 避免约束类型标记：设计的指令不需要标记上面知识体系中的结构类型或约束类型，要以真实自然的用户写指令的形式来构建，以上类别区分仅在于引导你如何思考和设计指令，但不要出现在设计的指令中。
6. 指令中客观具体约束不超过10个，并且尽量引导回复输出形式便于阅读，指令导向的回复不能过于简短。
7. 必须遵循：一次性多生成多个指令，如果指令字数<150字，输出5条构建指令；如果指令字数150-400字，输出3条评估指令；如果单个指令字数为400字以上，输出1-2条评估指令。

四、输出格式（严格遵守）

以 JSON 数组格式输出，每条指令包含 task_type 和 query 两个字段：

[
  {"task_type": "任务意图类型（如：信息抽取、标签分类等）", "query": "完整的指令内容"},
  {"task_type": "任务意图类型", "query": "完整的指令内容"}
]

字段说明：
- task_type：该指令对应的任务意图类型，如有多个意图用英文逗号分隔（如 "信息抽取,标签分类"）
- query：完整的指令正文，包含所有约束、素材、格式要求等内容

注意事项：
- 只输出 JSON 数组，不要有任何说明文字、前言或后记
- query 字段内的换行用 \n 表示，确保 JSON 格式合法可解析
- 不要在 JSON 外部添加任何标记或分隔符（不要用【$】等符号）

现在请根据可能输入的任务意图或领域，自行构思指令，如果没有任何输入，请你自行根据上面的任务意图定义，自由发挥设计指令。如果任务意图为其他，请你结合业务需求和逻辑，发散式构思其他可能的指令。
